## Summary

Add a mini RAG pipeline to NeuroDesk AI. Users upload PDF or TXT files; the system extracts text, chunks it into 512-token / 64-overlap segments (cl100k_base), generates 768-dim embeddings via Ollama `nomic-embed-text`, and stores them in `sqlite-vec` inside the existing `neurodesk.db`. When the user asks a question, the top 5 chunks are retrieved by cosine similarity and injected into the system prompt. `llama3.1:8b` generates a grounded, cited answer. A user-global document library persists across conversations. The existing chat API is extended, not replaced.

**Scale/Scope**: Single-user local tool, no authentication

## Constitution Check

### I. Modular Architecture ✅
New `src/modules/rag/` module, explicitly anticipated in the constitution's module map. Chat module calls RAG via `src/modules/rag/index.ts` public API. No circular imports.

### II. Test-First ✅
Failing tests written before implementation for: `pdf-extractor`, `chunker`, `embedding-client`, `retrieval-service`, documents API routes. Integration test covers full upload → retrieve → answer path.

### III. Security-First ✅
- MIME type + extension allowlist on upload
- File size enforced server-side (not trusting `Content-Length`)
- SHA-256 hash computed server-side; duplicate check blocks re-processing
- Files stored as UUIDs; original filename never used in filesystem path
- Retrieved chunks injected into system message (not user turn) to reduce prompt injection surface

### IV. API-First Design ✅
`src/app/api/documents/` routes (GET list, POST upload, GET detail, DELETE) defined with typed Zod contracts before UI components.

### V. Simplicity & YAGNI ✅
`sqlite-vec` eliminates need for a separate vector database. `pdf-parse` is the minimal extraction lib. Ollama generation reuses `createOpenAI` with custom `baseURL` — no new SDK. Cloud embedding fallback deferred to post-v1.

### VI. Observability & Documentation ✅
`src/modules/rag/index.ts` gets standard comment block per constitution. Processing status + error messages stored in DB. INFO logging on upload/completion; ERROR logging on extraction/embedding failures.

### VII. Incremental Delivery ✅
P1 (upload + answer) shipped first, independently deployable. P2 (multi-turn, provider switching) and P3 (citation UI panel) added as follow-on increments.

## Implementation Phases

### Phase 1 — Data Layer & Ingestion (P1 prerequisite)

1. Install `sqlite-vec`, `pdf-parse`, `@types/pdf-parse`
2. Extend `schema.ts` with `documents` + `document_chunks` tables
3. Extend `db/index.ts` to load sqlite-vec extension and create `vec_document_chunks`
4. Run `npx drizzle-kit push`
5. Implement `pdf-extractor.ts` — wrap pdf-parse, return `{ pageNumber, text }[]`
6. Implement chunker logic in `ingestion-pipeline.ts` — 512-token / 64-overlap sliding window using js-tiktoken
7. Implement `embedding-client.ts` — POST to Ollama `/api/embed`, return `number[]`
8. Implement `ingestion-pipeline.ts` — orchestrate extract → chunk → embed → insert
9. Implement `document-service.ts` — CRUD, SHA-256 dedup, status transitions
